## 1. 핵심 시대 구분

| Phase | 시기 | 무엇이 핵심이었나 | 대표 도구 |
|---|---|---|---|
| 1. **Prompt** | 2023 | 자연어로 의도와 제약을 전달 / 답변 품질 조정 | ChatGPT, Claude, Gemini |
| 2. **Context** | 2025 | 코드베이스·문서·브라우저 상태를 맥락으로 연결 | Cursor, Manus |
| 3. **Harness** | 2026 | 파일 수정·명령 실행·테스트를 한 루프로 / 반복 개선 | Claude Code, Codex |

> 가계부 적용: 이 프로젝트는 **Phase 3 (Harness)** 패러다임에서 작동하도록 설계되어야 한다.
> 즉 단발성 프롬프트가 아니라 plan→act→verify→loop 가 자동으로 돌아야 한다.

## 2. Agent / Skills / Harness 는 다르다

| 구분 | 비유 | 책임 |
|---|---|---|
| **Agent** | 두뇌 | 목표·제약 해석, 계획·우선순위, 결과 보고 후 전략 조정 |
| **Skills** | 손과 발 | 검색·파일 처리·API 연동 / 스크립트·템플릿 묶음 / 반복 작업 일관 수행 |
| **Harness** | 신경계 | 허용 도구·안전 경계, 호출 순서·재시도, 대화·메모리·상태 추적 |

> Garry Tan (Y Combinator CEO) 인용: *"Thin harness, Fat skills is a useful idea for AI agent."*
> 가계부 적용: `.claude/agents/*.md` 가 Agent 정의, `harness/` 가 Harness, `src/` + `scripts/` 가 Skills.
> 현재 구조와 일치한다 → 보강 방향만 정하면 된다.

## 3. 하네스 4요소 (PDF의 메인 메시지)

PDF는 하네스에서 4가지에 집중하라고 한다.

### 3-1. Tools (도구 관리)
- 정의: 반복 작업을 코드·API 호출로 묶어 만들고, 필요할 때 안정적으로 실행
- 왜?: 검색·계산·파일 처리처럼 흔들리는 작업을 빼놓고 일관 처리
- 설계 포인트: **입력·출력·부수효과 처리 규칙을 정해 도구 호출을 예측 가능**하게

### 3-2. Action Loop (액션 루프)
- 정의: 계획·실행·확인을 반복하며 실제 작업을 끝까지 진행
- 왜?: 루프가 촘촘할수록 복잡 태스크를 나누고 결과 품질을 끝까지 끌어올림
- 설계 포인트: **검토 기준과 재시도 조건을 두고 매 단계에서 다음 행동 결정**

### 3-3. State Management (상태 관리 및 평가)
- 정의: 진행 단계·완료 여부·평가 결과를 추적해 현재 상태를 종합
- 왜?: 완료 기준과 품질 기준을 분명히 해 성공한 완료만 인정
- 설계 포인트: **평가 결과를 기록하고 개선 지점·다음 우선순위로 연결**

### 3-4. Memory (메모리 관리)
- 정의: 사용자 선호·진행 피드백·산출물 결과를 저장하고 재사용
- 왜?: 작업 연속성·개인화를 높이고 유지할/버릴 정보 구분
- 설계 포인트: **검색·갱신·재사용이 쉬운 계층화 파일 구조**로 관리

> 가계부의 `analysis_cache`, `merchant_learning_rules`, `correction_logs` 가 이미 Memory 의 일부.
> Action Loop 는 candidate→approve 루프, State 는 Phase별 IMPLEMENTATION_STATUS 가 일부 담당.
> Tools 는 `services/*` 가 담당. 이미 4요소 형태가 어느 정도 갖춰진 셈.

## 4. 3계층 모델 (Runtime / Capabilities / Safety)

PDF의 도식 — 안에서 바깥으로:

```
[ Safety & Scale ]   ← Verification Loop, Guardrails, Tool Scoping, Subagent Orchestration
   [ Capabilities ]  ← Tools, Memory, Context Management, State Management
      [ Runtime ]    ← Prompt Construction, Orchestration Loop, Output Parsing, Error Handling
         ( LLM )
```

| 가계부의 어디에 해당 | 매핑 |
|---|---|
| Runtime | `lib/ollama/client.ts`(orchestration), 프롬프트 + zod 검증(parsing), try/catch(error) |
| Capabilities | `services/*` (tools), `analysis_cache`/`learning_rules` (memory), `lib/*` (context) |
| Safety & Scale | `audit-rls.mjs`/RLS(guardrails), `.claude/agents/*` Scope(tool scoping), CONTRACT.md |

## 5. 1인 빌더에게 왜 하네스가 중요한가

PDF 핵심 명제: **"1인으로는 불확실성을 완전히 제거해야 합니다."**

세 가지 압박:
- **너무 많은 할 일**: 기획·개발·세무까지 혼자 → 동시 작업을 나눌 구조 필수
- **모니터링의 한계**: 매번 사람이 개입하면 속도 저하 → **검증 루프로 자동 확인**
- **반복 업무의 늪**: 템플릿화 + 스킬 모듈화로 처리 시간 단축

> 가계부 적용: 사용자 본인이 PT·도수치료 본업 + 1인 창업이라는 점에서 정확히 일치.
> 그래서 `qa-harness` 가 read-only 자동 회귀로 분리된 것이 옳다.

## 6. 태스크별 다른 하네스 (PDF p17)

PDF는 4개 태스크 타입을 예시로 든다 — **각각 MODEL/GOAL/TOOLS/LOOP/MEMORY 가 다 다르다**:

| 축 | /dev (개발) | /ops (세무·부가) | /deck (자료 제작) | /post (콘텐츠) |
|---|---|---|---|---|
| MODEL | 코드 수정에 강한 모델 | 보수적 모델 (실수 비용 큼) | 디자인 + 텍스트 | 문장 흐름 |
| GOAL | 안정적 코드 수정 | 정확한 숫자/규정 | 슬라이드 산출 | 콘텐츠 배포 |
| TOOLS | Git, 테스트, 서브에이전트 | Drive, Sheets, OCR, 매핑 룰 | Drive, Docs, Slides 빌더 | 배포 채널, 미디어 |
| LOOP | 작성 → 실행 → 테스트 → 커밋 | 추출 → 검증 → 매뉴얼 확인 | 초안 → 사전 테스트 → 인간 확인 | 후보 발굴 → 작성 → 게시 |
| MEMORY | 리포 히스토리, 작업 메모 | 사업/세무 기준 | 디자인·구조·재사용 자료 | 본인 채널 톤 |

> 가계부 적용: 우리의 5개 에이전트가 정확히 이 패턴.
> `ai-extraction` 은 /ops 성격(보수적, 검증 무거움), `finance-core` 는 /dev 성격(CRUD 빠른 루프),
> `ux-design` 은 /deck 성격(시각 검증), `qa-harness` 는 자동 회귀 루프, `collab-security` 는 가장 보수적.

## 8. 에이전트 사용 시 반드시 피해야 할 4가지 패턴 (PDF p22)

| # | 패턴 | 한 줄 |
|---|---|---|
| 01 | **추상적 코드 작성 줄이기** | `useEffect` 등 높은 추상화 수준의 코드는 없앤다 |
| 02 | **불확실한 fallback 금지** | 실패를 모면하기보다는 **이유를 설명**해야 한다 |
| 03 | **검증 없는 종료 막기** | 하네스가 검증 절차를 강제 |
| 04 | **기능 경계 명확히** | 수정 범위·도구 권한·승인 지점을 미리 정함 |

> 가계부 적용 메모: CONTRACT.md §7-2 의 "환경변수 누락 시 graceful fallback 금지" 가 02번과 직접 일치.
> 4번은 `.claude/agents/*.md` 의 `Scope` / `Forbidden` 섹션이 이미 강제 중.
> 1번과 3번은 **추후 점검 항목** — 현재 코드에 useEffect 과한 추상화가 없는지, verify 없이 끝나는 흐름이 없는지.

## 9. 프로덕션 레벨 5가지 원칙 (PDF p24, Conclusion)

1. **필요한 업무를 모두 파악하라** — 에이전트가 맡을 일과 사람이 승인할 일을 나누고, **자동화하지 않을 범위**까지 먼저 정한다
2. **최대한 예측 가능하게 만들어라** — 입력 형식과 완료 기준을 고정, 실패 처리 규칙까지 명확히 남긴다
3. **오케스트레이션은 하나로 통합** — 도구·모델이 여러 개로 나뉘더라도 작업 지시·상태·로그는 한 곳에서
4. **모델에 종속되지 않게 설계** — 모델이 바뀌어도 목표·도구는 유지, 검증 루프가 같은 방식으로 돌게
5. **기본 인프라는 지속적으로 관리** — 메모리·컨텍스트·권한·템플릿은 한 번 만들고 끝이 아니라 계속 정리

> 가계부 매핑:
> 1) `.claude/agents/*.md` 의 Forbidden + 사람 승인 트리거가 이 역할
> 2) zod 검증 + verify 파이프라인 일부 충족, 더 강화 가능
> 3) 현재 Claude Code 단일 오케스트레이션 ✓
> 4) Ollama → OpenAI Vision 전환 사례에서 **검증 루프는 그대로 유지** 한 점이 4번 원칙 적용 사례
> 5) `consolidate-memory` 스킬 + RAG 재인덱싱이 5번에 해당
