## 목적

본업(PT/도수치료) 중에도 가계부 잡(영수증 1장 분석, 일괄 승인 알림, 예산 임계 도달 알림 등)을 **사람 모니터링 없이** 처리하거나, 사람 개입이 필요한 시점에만 모바일로 알림을 받기 위함.

**가계부는 이미 Vercel + Supabase 위에서 도는 웹앱**이고, Vercel cron 도 운영 중이다. 새 인프라를 크게 추가하지 않고 작동하는 디자인으로 접근.

---

## 두 갈래의 트리거 (분리 설계)

### A. 사람 → 가계부 (수동 트리거)
사용자가 모바일에서 영수증 사진을 보냈을 때 분석되어야 함.

**이미 있는 흐름**: PWA 카메라 업로드.
→ 신규 채널 도입 전에 **PWA 단축 아이콘 / 공유 시트 등록**으로 충분한지 먼저 확인.

추가 채널 후보 (필요 시):
- **이메일 → 업로드**: 영수증 사진을 특정 이메일로 보내면 inbound webhook 으로 받아 `uploaded_files` insert. (Postmark / Resend inbound 또는 Cloudflare Email Routing 활용)
- **카카오 비즈채널 / Telegram 봇**: 사진 업로드 후 토큰 매핑된 사용자의 파일로 저장.

> ⚠️ 이메일/메신저 inbound 는 **새 외부 의존성**이라 CONTRACT §3-3 (시크릿) / §6-3 (외부 관계) 검토 필수.

### B. 가계부 → 사람 (자동 알림)
시스템이 사람의 결정을 기다리는 상태가 되면 모바일로 push.

**이미 있는 흐름**: in-app notifications + 이메일(설정 시).
**부족한 부분**: 모바일 푸시(웹 push 또는 메신저 push).

후보:
- **PWA Web Push**: 표준이지만 iOS Safari 호환성 점검 필요. (현재 PWA 인프라 위에 push key 만 추가하면 됨)
- **Telegram 봇 푸시**: bot token + 사용자별 chat_id 매핑. 가벼움. 토큰 노출 시 즉시 회전.
- **Slack incoming webhook**: 본인 워크스페이스만 쓰면 1인용으로 충분.

## 도입 시 따져야 할 5가지 (CONTRACT 매핑)

| 점검 | 관련 조항 |
|---|---|
| 사용자 이름·전화·이메일 등 PII 가 외부 채널 메시지에 평문으로 들어가지 않는가 | §3-1 최소 수집 |
| 봇 토큰 / API 키 / inbound webhook 시크릿 어디에 저장하나 | §3-3 시크릿 관리 |
| 사용자 동의가 필요한 경우(개인정보 외부 채널 송신) UI에서 받는가 | §6-2 사용자 동의 |
| 모바일 채널이 다운될 때 가계부 본 기능이 안 망가지는가 (그 기능은 원래 in-app 으로 닫혀 있어야) | §4-1 핵심 구조 |
| 자동 의사결정(자동 승인 등)을 외부 채널 입력으로 트리거하지 않는가 | §2-3 자동 차단 / §5-2 최종 책임 |

## 권장 단계 (작은 단위로 도입)

1. **PWA Web Push 만 먼저**: 새 외부 채널 없이, 본인 기기에 기본 알림. (가장 보수적)
2. 부족하면 **Telegram 봇 푸시 (out-bound 만)**: 알림만 보내고, 사용자 응답으로 가계부를 조작하지 않음.
3. 더 필요하면 **Telegram inbound (영수증 업로드)**: 토큰 매핑 + 마스킹 + 7일 폐기 정책 그대로 적용.
4. **이메일 inbound 는 가장 마지막** (스팸/스푸핑 위험이 가장 큼).

각 단계는 [`docs/execute-plans/YYYY-MM-DD-mobile-trigger-N.md`](../../docs/execute-plans/) 으로 plan-first 진행.

## 의존성 (도입 결정 후)

도입 시점에 추가가 필요한 외부 의존성 — **현재는 추가하지 않는다**:

| 단계 | 후보 패키지/서비스 | CONTRACT 검토 |
|---|---|---|
| Web Push | `web-push` (Node) + VAPID 키 | §3-3 (VAPID 비공개 키 보관) |
| Telegram out | `node-telegram-bot-api` 또는 fetch | §3-3 (bot token 보관) |
| Telegram inbound | + Cloudflare Workers / Vercel Edge | §6-3 (3rd party 약관 / 학습 옵트아웃) |
| 이메일 inbound | Postmark inbound 또는 Cloudflare Email Routing | §6-3 + §1-1 (마스킹 통과 후만 저장) |

## 비-목표 (이 가이드에서 다루지 않는 것)

- AI 자동 승인 (CONTRACT §2-1 자동 승인 모드 부재)
- 외부 메신저로 거래 명세 평문 전송 (마스킹 후만 가능)
- 가족 멤버 간 푸시 권한 위임 (별도 설계 필요)
- 결제 / 알림 푸시 사용자 비용 분담 (B2C 무료 티어 한도 내에서만)
