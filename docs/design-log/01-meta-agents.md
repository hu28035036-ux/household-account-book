# Design Log 01 — 메타 에이전트 6개

흐름의 1·2·3·5·6·7 단계. 영역 작업을 직접 하지 않고 흐름을 제어한다.

각 항목 형식: **무엇** / **왜 이렇게 설계했나** / **대안과 그 기각 이유** / **실패 신호 (이 에이전트가 잘못 작동하면 어떻게 보이는가)**.

---

## conductor (1단계 · 총괄)

**무엇**: 사용자 입력을 받아 의도 분석 → Orchestrator에 작업 카드 송출. 흐름 종료 후 사용자 보고.

**왜**: PDF p15 *"1인으로는 불확실성을 완전히 제거해야 합니다"* 와 직접 대응. 모호한 입력이 영역 에이전트로 그대로 흘러가면 잘못된 코드를 만든 뒤 환송하느라 시간만 쓴다 — **모호함은 시작점에서 잡는다**.

**대안과 기각**:
- "Orchestrator가 의도 분석도 같이": 분배 + 의도 추론을 한 명에게 맡기면 책임이 흐려진다. 사용자 대화 책임과 기술 분배 책임은 다른 근육.
- "메인 세션이 conductor 역할 자동 흡수": 가능하지만 명시적 정의 파일 없으면 다음 세션이 같은 패턴으로 안 함. **명시 > 암묵**.

**실패 신호**:
- 같은 모호함을 두 번 묻는다 → 1회 한도 위반
- 의도 분석 없이 곧장 영역 에이전트 호출
- 종료 보고에 무엇이 바뀌었는지 누락

---

## orchestrator (2단계 · 분배)

**무엇**: Conductor 의 작업 카드 → 영역 매핑 / 병렬·직렬 결정 / plan-first 트리거.

**왜**: PDF 5원칙 3번 *"오케스트레이션은 하나로 통합"*. 영역 매핑을 사람이 매번 결정하면 일관성 깨진다. 같은 작업이 매번 다른 영역 조합으로 분배되면 학습이 안 됨.

**핵심 결정**:
- collab-security 는 **무조건 단독** — 다른 영역과 병렬 시 마이그레이션 번호 충돌 / RLS 변경이 다른 영역 import 깨뜨림
- qa-harness 는 **항상 read-only 병행** — 다른 영역 작업 중에 회귀 케이스 미리 누적
- 의존 순서: 스키마 → services → UI (역순 분배 금지)

**대안과 기각**:
- "에이전트가 자기 영역 결정": 영역이 겹치는 작업에서 두 에이전트가 같이 손대는 충돌 → 단일 분배자가 깔끔.
- "사용자가 매번 분배": 1인 빌더 시나리오에서 모니터링 비용 부담 — 자동화 핵심.

**실패 신호**:
- collab-security 를 다른 영역과 병렬로 묶어 dispatch
- plan-first 트리거 무시 (마이그레이션·시크릿 작업을 plan 없이 영역 분배)
- 의존 역순 분배

---

## sentinel (3단계 · 사전 가드)

**무엇**: 분배 카드를 받아 CONTRACT 위반 가능성 / 자동 진행 금지 항목 분류. 위반 시 conductor 환송.

**왜 추가했나** (사용자 흐름엔 없었음):
사용자 4단계 흐름은 사후 검증(verifier)만 있었음. 그 경우:
1. 위험 작업 (RLS 약화, 시크릿 노출 가능 코드, 마이그레이션) 이 일단 영역 에이전트에 들어감
2. 영역 에이전트가 작업을 시작 — 일부 이미 수정됨
3. Verifier 가 사후 발견 → 환송 → 되돌리기 비용

비가역 작업(마이그레이션 적용, 시크릿 커밋 등)은 사후 환송으로 회복 안 된다. **사전 차단이 유일한 안전선**.

**대안과 기각**:
- "Verifier가 사전·사후 둘 다": 한 에이전트가 분배 전·후 모두 검사하면 책임 모호. 사전(차단)과 사후(검증)는 의사결정이 다르다.
- "각 영역 에이전트의 Forbidden 으로 충분": Forbidden 은 작업 중 자제 룰이지, **작업 시작 차단 메커니즘이 아님**. Sentinel 이 시작 자체를 막는 것은 별개의 강제력.

**실패 신호**:
- RLS 변경 작업이 사용자 승인 없이 collab-security 로 dispatch
- 시크릿 노출 의심 코드가 차단 없이 영역 진입
- 차단 사유를 conductor 에 모호하게 전달 ("위험합니다" 식)

---

## verifier (5단계 · 1차 빠른 게이트)

**무엇**: 영역 에이전트 작업 종료 후 1차 검증 — typecheck / vitest / harness/run / smoke / RLS audit. 첫 실패에서 즉시 break.

**왜 1차로 두었나**:
- 큰 결함(타입 에러, 단위 테스트 실패, 명백한 위반)은 5회 반복 게이트 들어가기 전에 빠르게 차단
- loop-validator 가 5회 모두 돌면 25회 검증 — 기본 결함 잡는 데 25회는 낭비
- Verifier 가 통과한 작업만 loop-validator 로 진입 → 의미 있는 안정성 게이트

**대안과 기각**:
- "Verifier 없애고 loop-validator 만": 첫 회차에 잡힐 결함도 5회 카운터 사이클을 한 번 돌게 됨. 비효율.
- "Verifier 가 5회 반복": 사용자가 별도 에이전트로 추가 요청. **별도 에이전트 분리가 명령**.

**실패 신호**:
- 첫 실패에서 break 안 하고 다음 stage 계속 (CONTRACT §9-A-3 위반)
- pass 신호를 curator 로 직접 (loop-validator 건너뛰기) → 흐름 위반

---

## loop-validator (6단계 · 5회 연속 강제)

**무엇**: Verifier 통과 후 5회 연속 그린을 강제. 각 회차 점진 누적(typecheck → vitest → harness → smoke → RLS+responsive). 어느 회차든 실패 = 카운터 리셋 + 영역 환송.

**왜**: 사용자 명시 명령 — *"무조건 5회 검증, 각회마다 오류 발견 시 수정 후 다시 1회부터"*. 추가 동기:

- 한 번 통과 = 우연 가능. 5회 연속 = 안정성 신호.
- flaky(간헐 실패) 검출 — 같은 코드를 여러 번 돌리면 비결정성이 드러남
- Action Loop(PDF p11) 의 가장 엄격한 구현

**5단계 회차 모드 — 각 회차 의미를 갖도록**:
- 1: typecheck + vitest (가장 빠름)
- 2: + harness/run --mock
- 3: + ESLint + §9-A grep
- 4: + smoke:all
- 5: + audit-rls + responsive

같은 검증 5회는 비효율 → 점진 누적. 5회차 통과 = 모든 단계가 안정.

**대안과 기각**:
- "3회 / 7회": 사용자가 5회로 명시. 변경 시 본 문서 + loop.mjs `ROUNDS` 동시 갱신.
- "각 회차 같은 검증": 비효율 + 5회의 의미가 단순 회수만 됨. 변형 게이트가 더 많은 결함을 잡는다.
- "실패 시 다음 회차 진행": 카운터 누적이 아니라 리셋 — 사용자 명령 강제.

**한도 규칙**:
- 환송 사이클 5회(누적 25회 검증) 도달 시 conductor 보고 — 작업 의도 재검토 신호
- **차단 아님 — 신호 의무**. 사용자가 "계속" 또는 "중단" 결정.

**실패 신호**:
- 4회 통과 후 종료
- 실패 시 다음 회차 강행
- 같은 작업이 한도 도달했는데 계속 자동 진행

---

## curator (7단계 · 사후 큐레이션)

**무엇**: loop-validator 5회 연속 통과 후 — RAG 재인덱싱 / execute-plans 결과 기록 / MEMORY 갱신(명시 조건 시) / 후속 권고.

**왜 추가했나** (사용자 흐름엔 없었음):
PDF 5원칙 5번 *"기본 인프라는 지속적으로 관리"*.

Verifier 가 pass 만 띄우고 끝나면:
- RAG 인덱스 stale → 다음 세션이 새 문서 검색 못 함
- execute-plans 결과 미기록 → 작업이 무엇을 남겼는지 모름
- 회귀 케이스 추가 권고 누락 → 같은 결함 재발

**Curator = 인프라 단일 책임자**. 매번 사람이 챙기지 않아도 자동 관리됨.

**대안과 기각**:
- "qa-harness 가 RAG 도 함께": qa-harness 는 도구를 만드는 곳, Curator 는 도구를 **이번 작업** 에 적용하는 곳. 책임 다름.
- "각 영역 에이전트가 자기 작업 후 인덱싱": 에이전트마다 다른 시점에 빌드 → 일관성 깨짐. 단일 책임자가 깔끔.

**실패 신호**:
- RAG 인덱스 미갱신
- execute-plans 결과 섹션을 거짓으로 채움 (verify 실패였는데 pass 기록)
- 사용자 메모리에 추측 저장 (명시 조건 외)

---

## 회고용 메모

- 7단계는 무거워 보이지만 자명한 변경(타이포)에는 [3] sentinel pass + [4] 영역 + [5] verifier pass + [6] round 1만 + [7] curator 로 단축 가능
- 메타 단계 1·2 가 사실상 메인 세션 책임 — 별도 정의 파일은 *컨벤션 명시* 가 주 목적이지 자동 호출 강제 아님
- 단계 추가는 비싸다. 8단계로 늘리는 결정은 명백한 사고 또는 새 도메인 위험 신호 후에만.
