# Design Log 04 — RAG

`docs/` 30+ 문서 + AGENTS/CONTRACT + .claude/agents/* + harness 참고자료를 에이전트가 빠르게 검색할 수 있는 경량 인덱스.

---

## 1. 왜 임베딩 안 쓰나 (1단계 의도)

**현재 구현**: BM25-ish + 한국어 + 영문 토크나이저 + 제목/헤딩 가중치.

**임베딩을 처음에 안 쓴 이유**:
- 외부 의존성 0 (가계부 CONTRACT §6-3 의 외부 관계 회피)
- API 키 / 비용 / 네트워크 의존성 없음
- 가계부 docs 는 **고정 어휘** (RLS, 마스킹, household, OCR, candidates 등) — 키워드 매칭이 실용
- 빌드 시간 ms 단위 — 작업 끝날 때마다 부담 없이 재인덱싱

**임베딩 도입 트리거 (미래)**:
- docs 가 100+ 로 늘고 키워드 추론이 어려워질 때
- 유사 의미 검색이 필요해질 때 (예: "거래내역 페이지" ≈ "트랜잭션 화면")
- 비용/약관 검토 (CONTRACT §6-3) 후

도입 시점엔 `--embed` 모드로 확장 — 현재 BM25 와 공존.

---

## 2. 토크나이저 선택

### 한국어 형태소 분석기 안 씀
이유:
- 노리 / mecab / khaiii 모두 외부 의존성
- 가계부 docs 는 신조어 / 고유명사 (가계부 도메인) 가 많아 형태소 분석 실패율 높음
- BM25 는 토큰이 정확한 형태소가 아니어도 작동

### 음절 단위 + 영문 단어
구현 (`rag/lib/tokenize.mjs`):
- 한글 음절(`가-힣`) 그대로 토큰 (형태소 분리 안 함)
- 영문 단어 lowercase + 길이 ≥ 2
- 한국어/영문 stopword 최소 셋
- 기호로 분리

**왜**:
- "RLS", "household", "OCR" 같은 영문 코드 용어는 정확 매치
- "가족 공유", "마스킹" 같은 한국어는 어휘 단위로 매치
- 형태소 변형(공유 / 공유한다 / 공유는) 은 BM25 가 얕게라도 처리

## 3. 점수 모델 — BM25-ish

### 표준 BM25
```
score = Σ idf(q) · (tf · (k1+1)) / (tf + k1·(1 - b + b·dl/avgdl))
```
파라미터: `k1=1.5`, `b=0.75` (일반적 기본값).

### 가중치 추가
- 제목 토큰 매치: +1.5
- 헤딩 토큰 매치: +0.5

**왜**:
- 문서 본문에 우연히 포함된 토큰보다 제목/헤딩에 있는 토큰이 의미 강함
- 가계부 docs 는 헤딩이 명확한 구조 (PDF/Markdown) → 활용 가치 높음

## 4. 인덱싱 대상의 의도적 제한

### 포함 (`rag/lib/walk.mjs` INCLUDE)
- `AGENTS.md`, `CONTRACT.md`, `README.md`
- `docs/**/*.md`
- `.claude/agents/*.md`
- `harness/README.md`, `harness/references/*.md`
- `rag/README.md`

### 제외
- `node_modules/`, `.next/`, `dist/`, `build/`
- `test-results/` (자동 생성)
- 코드 파일 (.ts/.tsx/.js — 검색은 grep 으로 충분)
- 사용자 메모리 (`~/.claude/.../memory/`) — 영역 분리

**왜 코드 파일 안 넣나**:
- RAG 는 *설계 의도 / 정책* 검색이 1순위 — 구현 검색은 grep / IDE
- 코드까지 넣으면 토큰 폭발 + 점수 노이즈 ↑

**왜 사용자 메모리 안 넣나**:
- 메모리는 사용자 자산 (CONTRACT 외 영역)
- 검색 결과에 노출되면 의도치 않은 컨텍스트 누출 가능

---

## 5. 인덱싱 갱신 정책

### 누가 책임지나
**Curator (메타 흐름 7단계)**. 검증 통과 후 무조건 재인덱싱.

### 왜 단일 책임자
- 영역 에이전트마다 빌드하면 빌드 시점 분산 → 일관성 깨짐
- 빌드 누락 시 다음 세션이 stale 인덱스 사용 → 잘못된 검색 결과
- 단일 책임자가 매번 빌드 = 항상 최신

### sanity check
재인덱싱 후 *이번 작업 주제 키워드* 로 검색해 새 문서가 잡히는지 확인.
안 잡히면 INCLUDE 패턴 누락 / 빌드 실패 의심.

---

## 6. 검색 사용 패턴 (에이전트 입장)

### 작업 시작 시 컨텍스트 끌어오기
```bash
node rag/search.mjs "<관련 키워드>"
```
상위 3-5 결과를 Read 도구로 직접 열어 읽음.

### 영역 에이전트의 "Read first" 와의 관계
- 각 에이전트의 `# Read first` 섹션이 **고정 우선순위 문서** 목록
- RAG 검색은 **상황 의존 추가 컨텍스트** — 사용자 요청에 따라 다름

둘은 보완관계 — Read first 가 base, RAG 가 보강.

---

## 7. 알려진 한계

- **숫자/특수기호 검색 약함**: tokenize 가 기호 분리 → "v1.3.2" 같은 버전번호 검색 불가
- **한국어 어절 변형 미흡**: "공유한다" / "공유했다" 가 다른 토큰 — BM25 점수 분산
- **의미 검색 불가**: "사용자 지운다" / "계정 삭제" 같은 다른 표현은 별도 토큰
- **`global_learning_rules` 검색 시 의미 폭발**: 가계부에선 흔한 키워드 → 점수 신뢰도 ↓

이 한계들은 **임베딩 도입 시 해결** 후보. 현 단계엔 의식적 수용.

---

## 8. 비-목표

- 검색 결과 자동 LLM 요약 (검색은 Read 로 직접 보는 패턴)
- 사용자 질의 자동 확장 (synonym expansion)
- 다국어 (현재 한국어 + 영문 코드 용어만)
- 부분 인덱싱 / incremental update (전체 재빌드가 ms 단위라 충분)

---

## 회고용 메모

- 47–49 문서 / vocab 5천 / avgdl 500 단어 — BM25 가 잘 작동하는 규모
- 100문서 넘어가면 검색 결과 noise 늘어날 가능성 — 임베딩 도입 검토 시점
- 한국어 토크나이저는 음절 단위 — 어절 분리 도입 시 검색 정확도 ↑ 후보
