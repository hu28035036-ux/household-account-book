# Design Log 05 — 7단계 메타 흐름

```
[1] conductor → [2] orchestrator → [3] sentinel → [4] 영역 → [5] verifier → [6] loop-validator → [7] curator → 보고
```

이 문서는 흐름 자체의 설계 의도. 각 단계는 [01-meta-agents.md](./01-meta-agents.md) 참조.

---

## 1. 왜 7단계인가

### 너무 단순한 흐름의 문제
- 1단계 (사용자 → 에이전트 직접): 모호함 / 위험 / 검증 누락
- 3단계 (사용자 → 에이전트 → 검증): 사전 가드 / 안정성 / 인프라 갱신 누락

### 너무 복잡한 흐름의 문제
- 단계가 늘수록 유지비 ↑ / 사용자 인지 부담 ↑
- 자명한 작업(타이포)도 7단계 풀 패스 → 비효율

### 7단계의 균형
- 사전(1·2·3) / 작업(4) / 사후(5·6·7) 를 각 3단계로 균형
- 자명한 작업은 [3] sentinel pass + [4] + [5] 만 / 단축 흐름 허용
- 큰 작업은 [3] block → conductor 환송으로 미리 잡음

**핵심 명제**: 단계 추가는 비싸다. **8단계로 늘리는 결정은 명백한 사고 또는 새 도메인 위험 신호 후에만**.

---

## 2. 단계별 분기 표

| 단계 | 통과 → | 실패/차단 → |
|---|---|---|
| 1 conductor | orchestrator | 사용자 (1회 질문) / 종료 (단순 정보) |
| 2 orchestrator | sentinel | conductor (분배 불가) |
| 3 sentinel | 영역 에이전트 | conductor (사용자 승인 요청) / orchestrator (수정 권고) |
| 4 영역 에이전트 | verifier | (작업 자체 실패는 verifier 가 잡음) |
| 5 verifier | loop-validator | 영역 (수정사항 + 사유) |
| 6 loop-validator | curator | 영역 (카운터 리셋) / conductor (한도 도달) |
| 7 curator | conductor (보고) | (이 단계 실패는 거의 없음 — 인프라 갱신만) |

### 환송 = 단방향 흐름의 명시 분기
- "역순"이 아니라 **명시된 환송 케이스**
- 환송 메시지는 형식 강제 (수정사항 + 사유 + 다음 행동)
- 모호한 환송 ("문제가 있어요") 금지

## 3. 자명한 작업의 단축 흐름 (인정)

다음 경우 풀 7단계 안 거쳐도 됨:
- 타이포 / 1줄 픽스 / 주석 추가
- 문서 갱신 (코드 영향 없음)
- 디버그 로그 정리

단축:
```
[1] (의도 명확) → [3] (위험 신호 0) → [4] (1명 영역) → [5] (typecheck + vitest 만) → [7] (RAG 재인덱싱)
```

[6] loop-validator 5회는 자명 작업에 과함 — sentinel 이 자명함을 분류해 단축 흐름 허용.

**판단 기준**: 작업이 비가역인가 / 다른 영역에 영향 주는가 / 사용자 승인 필요한가. 셋 다 no 면 단축.

## 4. 병렬과 순차 — 어떤 단계가 어떤 모드인가

| 단계 | 모드 | 이유 |
|---|---|---|
| 1·2·3 | **순차** | 의사결정 단계 — 의존성이 본질 |
| 4 영역 | **병렬 가능** (조건부) | ai-extraction ⊕ finance-core ⊕ ux-design ⊕ qa-harness(read-only). collab-security 는 단독. |
| 5·6·7 | **순차** | 검증/큐레이션 — 누적 책임 |

병렬은 영역 에이전트 한정. 메타 단계는 모두 순차 (단계가 곧 의존성).

---

## 5. 한도 / 시간 / 무한 루프 방지

### loop-validator 환송 한도
5사이클 (누적 25회 검증) 도달 시 conductor 보고. 차단 아닌 신호.

### conductor 모호함 질문 한도
1회. 두 번째부터는 reasonable assumption + 결과 보고로 환원.

### sentinel 차단 후 사용자 응답 한도
없음 (사용자 결정 대기). 단, 작업이 stale 한 채 24시간 이상이면 conductor 가 알림.

### Verifier 첫 실패 break
즉시. 다음 stage 안 돔 — 노이즈 줄이기.

### Curator 재인덱싱 시간
ms 단위 — 한도 불필요.

---

## 6. 흐름 위반 신호

다음이 보이면 **흐름이 잘못 작동**한다:

| 신호 | 의미 |
|---|---|
| 영역 에이전트가 sentinel 통과 없이 직접 작업 | [3] 우회 |
| Verifier 가 pass 인데 curator 로 직접 | [6] 건너뛰기 |
| loop-validator 4회 통과 후 curator | 5회 강제 위반 |
| Curator 가 verify 실패 작업 처리 | 흐름 역행 |
| 환송 메시지에 수정사항 없음 | 형식 위반 |
| 같은 모호함 두 번 질문 | conductor 1회 한도 위반 |

이 신호 발견 시 즉시 [`/harness/runbook.md`](../../harness/runbook.md) 에 사고로 기록.

## 7. 흐름 변경의 비용

흐름은 시스템 전체에 박혀 있다. 변경 시 갱신 대상:

- `AGENTS.md` §4 (흐름도)
- `.claude/agents/*.md` 의 Position in flow + Hand-off 섹션
- `docs/design-log/01-meta-agents.md` + `05-flow.md`
- 환송 메시지 형식이 달라지면 영역 에이전트의 *Verify before handoff* 갱신
- RAG 재인덱싱

이 비용 때문에 흐름 변경은 신중. 일반 보강(에이전트 1개 책임 보강)은 흐름 변경 아님 — 단계 추가/삭제만 흐름 변경.

---

## 회고용 메모

- 7단계는 1인 빌더에 무거워 보이지만, 단축 흐름이 인정되어 자명한 작업은 빠름
- 사용자가 "5회 검증" 명시 후 6단계가 7단계가 됨 — 다음 변경도 사용자 명령 기반이어야 함
- 흐름의 가장 약한 고리는 **환송 메시지 형식** — 모호하면 영역 에이전트가 무엇을 고쳐야 하는지 모름
